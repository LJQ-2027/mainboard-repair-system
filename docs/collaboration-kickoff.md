# 协作启动清单

## 1. 当前协作目标

本仓库将作为智能主板维修系统的产品原型和后续工程协作仓库。近期协作重点不是立即堆功能，而是先对齐产品方向、资料边界、第一阶段范围和工程结构。

## 2. 建议近期工作

### 2.1 资料边界清理

- 检查公开仓库中是否存在内部飞书链接或敏感资料线索；
- 将真实内部链接迁移到受控配置或内部版本；
- 公开仓库仅保留脱敏示例和通用说明。

### 2.2 产品需求对齐

- 确认第一阶段是否聚焦“知识库 + 诊断 SOP + 可视化操作工具”；
- 明确 AI 初期只做辅助导诊、检索和摘要，不做最终维修判定；
- 确认后续是否需要支持主板点位图、案例库、工单升级和专家协同。

### 2.3 工程结构整理

当前系统已经完成第一步结构整理：主页面仍保留为 HTML 原型，项目资料、位号映射、诊断规则、知识库和 log 规则已拆分到 `data/` 目录。短期可以继续保留原型形态，但建议逐步补充：

- `docs/`：需求、边界、协作说明；
- `examples/`：脱敏示例数据；
- `tests/`：后端转换逻辑和接口测试；
- `frontend/`：后续如需拆分前端 UI 和交互逻辑，可逐步迁移；
- `backend/`：后续如需扩展服务，可逐步迁移。

### 2.4 第一批可做任务

- 移除或配置化公开 HTML 中的内部链接；
- 梳理一个标准诊断 SOP 的数据结构；
- 增加一个脱敏机型/故障示例；
- 为 AI 代理服务补充基础异常处理和测试；
- 明确本地启动、API Key 配置和安全注意事项。

## 3. 建议协作方式

- `master` 保持可运行；
- 较大改动通过 Pull Request 讨论；
- 小的文档补充可以直接提交；
- 涉及资料边界、内部链接、真实案例、机型信息的改动，需要先确认是否可公开；
- AI 相关能力先以“辅助建议”定位，不在未验证阶段承诺自动维修结论。

## 4. 与售后能力建设项目的关系

售后侧已有一套“海外主板维修能力建设项目”的前期规划材料，重点关注：

- 为什么要降低不必要整板更换；
- 如何把专家经验转成可执行流程；
- 如何建立主板维修知识库和 SOP；
- 如何逐步走向 AI 赋能和软硬件自动化检测。

本仓库可以承接其中的系统原型、产品化实现和后续工程开发。
