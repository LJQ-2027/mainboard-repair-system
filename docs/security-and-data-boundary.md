# 安全与资料边界

## 1. 基本原则

主板维修系统会涉及工程资料、内部案例、维修经验和可能的机型信息。即使系统本身开源或对外协作，也应避免把内部敏感资料直接写入公开仓库。

建议原则：

- 公开仓库只保留产品代码、脱敏示例、通用说明和协作规范；
- 内部资料保留在公司授权的知识库、飞书空间或私有存储中；
- 代码中不要硬编码内部资料链接、真实机型敏感信息、测试数据或维修参数；
- API Key、账号、token、cookie 等凭证必须只放本地环境变量或受控密钥管理中。

## 2. 不建议放入公开仓库的内容

以下内容如需使用，应放在受控内部空间，并按权限管理：

- FCT/ICT 测试逻辑；
- FA 失效分析报告；
- BoardView、原理图、Layout 文件；
- Golden Sample 数据；
- 维修 SOP 原始文件、温度曲线和关键工艺参数；
- 坏板样本信息；
- 未发布机型资料；
- 成本、备件价格和内部权限材料；
- 内部飞书、网盘、知识库等资料链接；
- 任何真实 API Key 或访问凭证。

## 3. 当前公开仓库需要关注的风险

当前前端原型中存在若干内部飞书链接。由于仓库是 public，这些链接本身已经暴露在公开互联网上。即使外部人员没有权限访问内容，公开链接也可能暴露内部资料结构或业务线索。

建议后续处理方式：

- 将内部链接从前端代码中移除；
- 改为读取本地配置文件或后端接口；
- 公开仓库只保留脱敏示例链接或占位数据；
- 如需保留真实内部链接，应改用私有仓库或公司内部部署版本。

## 4. AI 使用边界

AI 输出只能作为辅助建议，不应作为最终维修判定。涉及主板维修动作、换件建议、维修边界和修后验证时，应以经过制造中心或技术支持确认的 SOP 和知识库为准。

AI 初期适合承担：

- 检索知识库；
- 解释 SOP；
- 汇总维修记录；
- 生成升级专家的工单摘要；
- 提醒遗漏检查项。

AI 初期不建议承担：

- 直接判定某颗元器件必须更换；
- 替代工程师定义维修边界；
- 在缺少测试数据时给出确定性结论；
- 基于未确认资料生成新的维修标准。

## 5. 后续建议

在继续开发前，建议先完成一次资料边界清理：

- 明确公开版、内部版各自包含什么；
- 移除公开代码中的内部链接；
- 提供 `.env.example`，但不提交真实 `.env`；
- 建立脱敏测试数据；
- 在 README 或 docs 中说明资料和 AI 输出的使用边界。
